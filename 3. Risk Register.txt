🧱 A. INTRODUCTION (FOR REPORT)

In any software project, identifying potential risks before development is critical to ensure smooth execution and successful delivery. A risk register helps in systematically analyzing possible issues, estimating their impact, and planning mitigation strategies.

For the SahayakAI project, risks were identified across technical, data, team, and deployment aspects to minimize failures and ensure a stable MVP within the given timeline.

---

📊 B. RISK REGISTER TABLE (MAIN CONTENT)

👉 Use this table in your report:

| Risk ID | Risk Description                     | Probability | Impact   | Reason                                       | Mitigation Strategy                               |
| ------- | ------------------------------------ | ----------- | -------- | -------------------------------------------- | ------------------------------------------------- |
| R1      | Incorrect eligibility logic          | High        | High     | Simplified conditions may give wrong results | Test with multiple cases, validate logic manually |
| R2      | Dataset inaccuracies                 | High        | High     | Manual data collection errors                | Cross-check with official government websites     |
| R3      | Invalid or broken links              | Medium      | High     | Wrong or outdated URLs                       | Verify links before final deployment              |
| R4      | Time constraint (1 week)             | High        | Critical | Limited development time                     | Strictly follow MVP (3 features only)             |
| R5      | Frontend-backend integration failure | Medium      | High     | API mismatch or data errors                  | Test APIs using Postman before integration        |
| R6      | Incorrect user input                 | Medium      | Medium   | Users may enter wrong format                 | Add input validation and constraints              |
| R7      | AI API failure (optional feature)    | Low         | Medium   | API downtime or rate limit                   | Use static fallback explanation                   |
| R8      | Team coordination issues             | Medium      | High     | Lack of task clarity                         | Use RACI matrix for clear role assignment         |
| R9      | Deployment failure                   | Low         | High     | Hosting/server issues                        | Test deployment early, keep backup                |
| R10     | Demo failure during evaluation       | Low         | Critical | Network/server crash                         | Keep screenshots and recorded demo ready          |
